# 작업: 공유기(AX3000BCM) 네트워크 개선 — 맥미니 세션

- **대상**: ipTIME AX3000BCM 메인 공유기 (192.168.0.1, admin 로그인)
- **작성**: 맥북 세션, 2026-05-16
- **수행 방법**: 맥미니 로컬 Claude 세션. 맥미니는 유선(en0)이라 WLAN 재시작 중에도 인터넷 무중단 → 라우터 작업 적격.
- 먼저 `context/machines.md` 와 `tasks/macbook.md`(네트워크 딥리뷰 결과)를 읽을 것.

## 배경

맥북 세션의 네트워크 심층 진단 결과, 회선 "속도"가 아니라 "지연"이 문제로 규명됨:

- **버퍼블로트** — `networkQuality` 부하 시 HTTP 왕복 35ms → **632ms**. 다운/업로드만 하면 다른 동작이 ~0.6초 밀림. 회선·라우터 큐 문제라 **모든 기기 공통**(맥미니·맥북·그램·아이폰·아이패드).
- **Wi-Fi 첫 홉 지터** — 라우터 ping stddev 25ms. 5GHz ch36(160MHz)에 -47dBm 동일채널 160MHz 이웃이 완전히 겹쳐 간섭.

PC 소프트웨어 쪽은 이미 최적(맥미니=유선+한국 보안 SW 비활성 완료, 맥북=이번 세션 점검·한국 SW 비활성 완료). **남은 소프트웨어적 개선은 공유기 설정 2개뿐.** 맥북에서 시도했으나 ① ipTIME 15.x `/ui/` SPA가 헤드리스 자동화에 부적합 ② 맥북은 WLAN 재시작 시 끊김 → 유선인 맥미니로 핸드오프.

## 작업 (192.168.0.1 브라우저 접속, admin 로그인)

맥미니는 GUI가 있으니 일반 브라우저로 192.168.0.1 열어 수동으로 하는 게 가장 빠름.

### B. QoS — 버퍼블로트 완화

- QoS / 트래픽 관리 메뉴 확인.
- WAN 총 대역을 캡할 수 있으면: 다운로드 **~70Mbps**, 업로드 **~43Mbps** (실측 81↓/49↑의 약 88%).
- ⚠ ipTIME QoS는 fq_codel/CAKE 같은 진짜 SQM이 아님. per-device 우선순위만 지원하면 버퍼블로트 효과는 제한적 → 그 경우 B는 건너뛰고 C에 집중. 효과는 주로 업로드측.

### C. 5GHz 채널 폭 160MHz → 80MHz  ★핵심

- 무선 설정(5GHz) → 채널 대역폭 → 160MHz를 **80MHz**로 변경.
- ch36에 -47dBm 동일채널 160MHz 이웃이 겹쳐 지터 유발. 80MHz로 좁히면 겹침·경쟁↓.
- 최고 PHY는 1800→900Mbps로 반감되지만 100M 회선엔 영향 0. 지터·안정성만 개선.
- ⚠ 적용 시 WLAN ~50-80초 재시작 — 맥북·그램·아이폰·아이패드 일시 끊김(맥미니 en0은 무중단).

## 검증

- 적용 전후 `networkQuality -v` 비교 — Responsiveness(RPM)와 loaded latency가 핵심 지표.
- `ping -c 25 192.168.0.1` 으로 stddev(지터) 비교.

## 범위 밖 (하드웨어 — 사용자 별도 판단, 이 task 아님)

- 맥북용 USB-C/썬더볼트 유선 어댑터, SQM 지원 공유기 교체, LG U+ 회선 업글(500M/1G).

## 결과

_맥미니 세션 수행, 2026-05-16 17:25~18:10 KST_

### C. 5GHz 채널폭 160 → 80MHz — 완료 ✅

라우터 192.168.0.1 → 관리도구 → 무선랜 관리 → WiFi 고급 설정 → 5GHz 채널 크기 `160`→`80` 적용.
WLAN ~65초 재시작 후 자동 복구. 적용 확인:
- 라우터 UI: 채널 크기 `80` 저장됨.
- 맥미니 en1: `Channel 36 (5GHz, 80MHz)`, Transmit Rate 1200→1020 (80MHz로 PHY rate 반감 — 예상된 것, 100M 회선엔 영향 0).

### B. QoS — 맥북으로 핸드오프 (이 세션 미적용)

QoS 메뉴 = Down/Up 총 대역폭 슬라이더(1~1000Mbps) + 스마트 QoS 실행/중단 + IP·프로토콜별 규칙 테이블. 현재 전부 꺼짐(규칙 0).
WAN 총 대역 캡은 기술적으로 가능하나, **맥미니는 인터넷이 en0(상위 공유기 유선)이라 AX3000BCM QoS 효과를 측정·검증할 수 없음**. QoS 수혜·검증 주체가 같은 무선 기기여야 하므로 `tasks/macbook.md`로 핸드오프(사용자 결정). 라우터 QoS는 손대지 않음 — 꺼진 상태 그대로.

### 측정 (맥미니 — en1 Wi-Fi ping)

| 지표 | Before(160MHz) | After(80MHz) |
|------|----------------|--------------|
| ping 192.168.0.1 지터 stddev | 0.364 ms | 0.397 ms |
| ping avg / max | 2.79 / 3.63 ms | 2.96 / 3.89 ms |

맥미니 en1은 baseline부터 0.364ms로 이미 최적 → 변경 후에도 동일. **맥미니는 채널폭 효과 검증 대상이 아님** — baseline 지터 25ms였던 맥북이 검증 대상. (networkQuality는 en0 경로라 채널폭과 무관해 비교 생략.)
※ 전환 직후 1회차 측정은 stddev 1.45ms로 튀었으나 무선 재안정화 후 0.40ms로 정상 복귀 — 전환 직후 노이즈.

### 결론

채널폭 변경은 무해하게 적용 완료. 맥미니 자체엔 영향 없음(정상). 실효과(무선 지터 감소) 검증은 맥북·그램 몫.
