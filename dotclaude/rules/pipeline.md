# Pipeline Rules

모든 프로젝트 공통. 작업의 생명주기를 4단계로 관리한다.

## 단계

```
Scope → Design → Implement → Close
```

## Scope (범위 확정)

"무엇을 할 것인가"만 정한다. 어떻게(how)는 금지.

- **핵심 기능 3개 이내** — 초과 시 작업을 쪼갠다
- **"내일 쓰는가" 테스트** — No면 제거
- **규모 판단**:
  - small (1세션): Design 생략, Scope→Implement 직행
  - medium (2-3세션): Design 필수
  - large (계획 문서 필요): Design 필수 + 설계 문서 작성

게이트 → Design:
- 핵심 기능 3개 이내
- 각 기능 "내일 쓰는가" 통과
- 규모 판단 완료

## Design (설계)

"어떻게 할 것인가"를 정한다. 코드 작성 금지.

- **프로덕트 마인드셋** — 프로토타입이 아니라 이대로 쓸 수 있는 것을 설계
- **핵심 집중** — 가장 중요한 기능 1개에 설계 시간의 50% 이상
- **기능 추가 = 기능 삭제 검토** — 새 기능이 떠오르면 기존 기능 하나를 삭제 후보로
- **구현 복잡도 역산** — "몇 파일, 몇 줄?" 추정 불가 = 설계 미완
- **설계 문서**: `~/.claude/plans/` 또는 프로젝트 `docs/specs/`

게이트 → Implement:
- 핵심 기능별 구현 경로 명확 (파일, 변경 범위)
- "뺄 수 있는 게 있는가?" 재검토 완료
- 설계 문서 저장됨

## Implement (구현)

설계대로 만든다. 범위 확장 금지.

- **설계 벗어나면 멈춤** — "이것도 하면 좋겠다" → work-log에 기록만
- **한 세션 = 한 작업 단위** — 끝나면 /clear, 이어가면 work-log에 기록
- **검증 포함** — 구현 완료 = 테스트/확인까지 완료

게이트 → Close:
- 설계 문서의 핵심 기능 전부 구현됨
- 각 기능 검증됨

## Close (마무리)

정리하고 다음을 위해 상태를 남긴다.

- work-log 갱신 — 완료한 것, 배운 것 1줄씩
- 메모리 갱신 — 다음 세션에 필요한 것만
- 설계 문서 상태 표시 — 완료/부분완료/보류
- 워크트리 사용했으면 정리 — 머지 또는 삭제
